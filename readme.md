## 🔧 Алгоритм работы

### 1. Инициализация (main.go)
- Обрабатывается аудиофайл в зависимости от того, в какой режим переключена система (f- режим обработки файла, r- режим обработки аудиострима)
- Из фаила или стрима по очереди копируются данные размером с `Chunk`
- Инициализируются два процессора: `beginning` (прямой порядок) и `inversely` (обратный порядок) для обработки скопированного.

### 2. Работа процессоров (beginning.go и inversely.go)

**Общая логика для обоих процессоров:**

## **Aлгоритм обработки Crumb**

Алгоритм обработки каждого Crumb состоит из нескольких ключевых этапов, обеспечивающих эффективное управление счетчиками и их значениями.

1. **Проверить сброс** (если `Value ≥ CounterValue` → делим все на 2).  
2. **Отсортировать счетчики** (по убыванию `Value`).  
3. **Проверить топ-N** (если совпадение → `PredictIncrement + Increment`).  
4. **Проверить остальные** (если совпадение → `Increment`).  
5. **Создать новый** (если нет совпадений → `Value = Increment`).  
6. **Сохранить состояние** (в `.bin` файл).  
7. **Фильтровать** (удалять редко используемые счетчики). 

---

### **1. Проверка условий сброса (Threshold Check)**
Перед обработкой Crumb система проверяет, не достигли ли какие-либо счетчики порогового значения `CounterValue` (из конфига).

- **Если какой-то счетчик ≥ `CounterValue`:**
  - Все счетчики делятся на **2** (целочисленное деление).
  - Это предотвращает переполнение и "затухание" старых данных.
  - Логируется событие сброса.

---

### **2. Сортировка счетчиков по убыванию значений**
Для оптимизации поиска система сортирует счетчики по **убыванию** их текущих значений (`Value`).

**Зачем?**  
- Быстрый доступ к "горячим" (часто используемым) счетчикам.
- Ускорение поиска совпадений в приоритетном порядке.

---

### **3. Приоритетная проверка топ-N счетчиков**
- **Сравниваем Crumb с индексом счетчика в пределах Actualization (`Counter.Data`, а не со значением)!**
- Пришел Crumb [0x01, 0x02] → система ищет счетчик с Data = [0x01, 0x02].
- Если неизменяемый `ID` = значению`Crumb` → `PredictIncrement + Increment`.
- Засчитывает совпадение.
- **Запись совпадений**:
   - При каждом совпадении данные записываются в отдельный файл `*_matches`
   - Формат записи: `[crumb (4 байта), ID (4 байта)]`

### **4. Проверка всех остальных счетчиков (включая топ-N)**
- **Повторно проверяем все счетчики** (включая те, что уже проверялись в топ-N).
- Если неизменяемый `ID` = значению`Crumb` → только `Increment`.

### **5. Создание нового счетчика (если совпадений нет)**

Если нет ни одного совпадения, создает новый счетчик с неизменюемым `ID` = текущего`Crumb`, с начальным значением `Increment`.

### **6. Сохранение состояния (в файл)**
После каждой модификации счетчиков система сохраняет их в **.bin файл** (временный файл → переименование).

---

### **7. Фильтрация счетчиков (по расписанию)**
Каждые **`FiltrationPeriod` чанков** система удаляет `FiltrationValue` самых маленьких счетчиков.

**Структура даннысчетчиков:**
   - Каждый счетчик занимает  байт:
     - 4 байта: индекс счетчика (присваивается при создании .bin)
     - 4 байта: значение счетчика
   - Максимальное количество счетчиков: 65536

**Работа с файлами:**

   - **loadOrCreate**:
     - Загрузка существующих данных из файла
     - Если файл не существует - создание нового с нулевыми значениями

   - **initializeFile**:
     - Инициализация нового файла данных
     - Заполнение нулями

   - **save**:
     - Сохранение текущего состояния всех счетчиков в файл
     - Гарантированная запись всех изменений

**Особенности реализации:**
   - Потокобезопасность (использование sync.Mutex)
   - Проверка целостности данных
   - Обработка ошибок ввода-вывода
   - Оптимизированное хранение данных

## 3 **Визуализация (simple_visualization.py)**
В `simple_visualization.py`:
- Чтение `.bin` файлов (`beginning.bin`, `inverse.bin`).
- Подсчет:
  - **Beginning**: данные значений счетчиков в прямом порядке (синий цвет).
  - **Inverse**: данные  значений счетчиков в обратном порядке (розовый цвет).
  Вот полная реализация с записью совпадений в отдельные файлы и их чтением для визуализации:

   - **Чтение для визуализации**:
   - Чтение из `*_matches.bin` файлов
   - Анализ количества и частоты совпадений

   - **Автоматическое создание файлов**:
   - Если файлов нет, они создаются при первом совпадении
   - Используется режим `os.O_CREATE`

   - **Совместимость**:
   - Сохранена обратная совместимость с существующей системой
   - Все оригинальные функции продолжают работать
