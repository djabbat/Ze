**Этот проект представляет собой систему обработки данных, которая:**
- Разбивает данные фаила или потока на блоки размером `Crumb` и создает `Chunk`
- Обрабатка ведется двумя способами:
- Прямой порядок (beginning).
- Обратный порядок (inversely).
- Сохраняет результаты в соответствующие beginning.bin и inverse.bin, доступную для API

## 🔧 Алгоритм работы

### 1. Инициализация (main.go)
- Обрабатывается аудиофайл в зависимости от того, в какой режим переключена система (f- режим обработки файла, r- режим обработки аудиострима)
- Из фаила или стрима по очереди копируются данные размером с `Chunk`
- Инициализируются два процессора: `beginning` (прямой порядок) и `inversely` (обратный порядок) для обработки скопированного.

### 2. Работа процессоров (beginning.go и inversely.go)

**Общая логика для обоих процессоров:**

1. **Инициализация:**
   - Проверка существования файлов данных:
     - `data/beginning.bin` для BeginningProcessor
     - `data/inverse.bin` для InverseProcessor
   - Если файлы отсутствуют - создаются новые размером `2^Crumb`, заполненные нулями

2. **Обработка данных:**
   - Данные поступают блоками `Chunk` через канал `dataChan`
   - Каждый блок разбивается на элементы `Crumb` размером `crumb_size`

3. **Различия между процессорами:**
   - **BeginningProcessor**:
     - Байты обрабатываются в прямом порядке (первый байт - старший)
   - **InverseProcessor**:
     - Байты обрабатываются в обратном порядке (последний байт - старший)

4. **Алгоритм обработки каждого Crumb:**
   - **Проверка на сброс**:
     - Если значение любого счетчика достигает `CounterValue`:
       - Все счетчики делятся на 2 (целочисленное деление)
   - **checkReset** - проверка условий сброса:
     - Если любой счетчик достиг порога `CounterValue`:
       - Инициировать сброс всех счетчиков
   - **resetAllCounters** - процедура сброса:
     - Все значения счетчиков делятся на 2 (целочисленное деление)

    - **Поиск совпадений**:
     1. Найти счетчик с максимальным значением в соответствующем `.bin` файле
     2. Сравнить значение индекса этого счетчика со значением текущего Crumb
        - При несовпадении: перейти к следующему счетчику с меньшим значением
        - При совпадении: 
          - Увеличить значение счетчика на `PredictIncrement`
          - Перейти к следующей проверке
   
   - **Актуализация значений**:
     - Сортировать значения счетчиков по убыванию
     - Проверить значение `ActualizationValue` (должно быть 0 < x < 1)
     - Количество итераций цикла вдоль сортированного списка счетчиков: `(размер Chunk / размер Crumb) * ActualizationValue`
     - В рамках цикла:
       - Сравнить значение Crumb со значениями всех индексов `.bin` файла сортированных по убыванию значений счетчиков
       - При полном совпадении:
         - Увеличить значение счетчика на `predict_increment`
   
   - **Формирование Chunk**:
     - Записать значение Crumb в текущий Chunk
     - Сдвинуть предыдущие записи (если есть) влево
     - Условия завершения:
       - Chunk полностью заполнен
       - Остаток Crumb (если не помещается) сохраняется для следующего Chunk
       - В файловом режиме: остаток дополняется нулями

5. **Завершение работы:**
   - При получении сигнала (Ctrl+C):
     - Остановка потока данных
     - Корректное завершение всех горутин
   - В файловом режим сохранение итоговых данных в соответствующие `.bin` файлы
   - В режиме потока бесконечно

### 4. Система счетчиков (counter_system.go)

1. **Структура данных:**
   - Каждый счетчик занимает  байт:
     - 4 байта: индекс счетчика (присваивается при создании .bin)
     - 4 байта: значение счетчика
   - Максимальное количество счетчиков: 65536

3. **Работа с файлами:**

   - **loadOrCreate**:
     - Загрузка существующих данных из файла
     - Если файл не существует - создание нового с нулевыми значениями

   - **initializeFile**:
     - Инициализация нового файла данных
     - Заполнение нулями

   - **save**:
     - Сохранение текущего состояния всех счетчиков в файл
     - Гарантированная запись всех изменений

4. **Особенности реализации:**
   - Потокобезопасность (использование sync.Mutex)
   - Проверка целостности данных
   - Обработка ошибок ввода-вывода
   - Оптимизированное хранение данных

### 3. Визуализация (simple_visualization.py)
- В фоновом режиме читает данные из beginning.bin и inverse.bin.
- Отображает 4 графика:
  - **Beginning**: данные в прямом порядке (синий цвет).
  - **Inverse**: данные в обратном порядке (розовый цвет).
- Обновляет графики каждую секунду.
- При получении сигнала завершения (Ctrl+C) останавливаются все процессы
